### 1.2 개발팀 운영 방식 (10명)

#### 팀 구성

| 역할 | 인원 | 주요 업무 | 보고 대상 |
|------|------|----------|----------|
| **시디 (팀장)** | 1명 | 작업 분배, 기술 리뷰, 산출물 검토, 팀 관리 | PM |
| **시니어 개발자** | 2명 | 대형/복잡 화면 분석, 백엔드 연동 분석, 멘토링 | 시디 |
| **개발자** | 5명 | 일반 화면 분석, 코드 리뷰, 문서화 | 시디 |
| **주니어 개발자** | 2명 | 팝업/단순 화면 분석, 문서 정리 | 시디 |

#### 운영 규칙

```
[개발팀 운영 원칙]

1. 작업 할당
   - 시디가 작업 분배 (PM 승인)
   - 능력/경험 고려한 할당
   - 일일 최대 2개 화면 제한

2. 작업 진행
   - 분석 템플릿 준수 (필수)
   - 진행 상황 실시간 공유 (Slack/Notion)
   - 어려움 발생 시 즉시 리포트

3. 산출물 관리
   - Notion에 문서 작성
   - GitHub에 코드/스크립트 저장
   - 산출물은 24시간 이내 업로드

4. 품질 기준
   - 체크리스트 100% 완료
   - 시디 리뷰 통과
   - 피어리뷰 통과
```

#### 시디의 역할 상세

| 역할 | 세부 업무 | 빈도 |
|------|----------|------|
| **팀장** | 일일 스탠드업 진행, 팀원 동기화 | 매일 09:00 |
| **기술 리더** | 복잡한 기술 문제 해결, 아키텍처 결정 | 수시 |
| **리뷰어** | 산출물 검토, 품질 관리 | 매일 |
| **리포터** | PM에게 진행 상황 보고 | 매일 18:00 |
| **에스컬레이터** | 블로커 이슈 PM에게 에스컬레이션 | 즉시 |

### 1.3 피어리뷰팀 운영 방식 (3명)

#### 팀 구성

| 역할 | 인원 | 주요 업무 | 보고 대상 |
|------|------|----------|----------|
| **피디 (팀장)** | 1명 | 리뷰 기준 정의, 품질 관리, React 설계, 리뷰 분배 | PM |
| **시니어 리뷰어** | 1명 | 복잡 화면 리뷰, 공통 컴포넌트 정의, 기술 가이드 작성 | 피디 |
| **리뷰 어시스턴스** | 1명 | 문서 품질 검토, 표준 준수 확인, 체크리스트 점검 | 피디 |

#### 운영 규칙

```
[피어리뷰팀 운영 원칙]

1. 리뷰 기준
   - 리뷰 체크리스트 기반 평가
   - Pass/수정요청/재검토 3단계 판정
   - 모든 화면은 리뷰 필수

2. 리뷰 프로세스
   - 개발자 → 자가 점검 → 시디 리뷰 → 피디 리뷰
   - 복잡 화면: 시니어 리뷰어 추가 검토
   - 리뷰 의견은 24시간 이내 피드백

3. 품질 관리
   - 주간 품질 메트릭스 작성
   - 반복적 오류 패턴 분석 및 개선
   - 개발팀 교육/가이드 제공

4. React 설계
   - 공통 컴포넌트 설계 및 문서화
   - 기술 스택 선정 및 검증
   - POC(Proof of Concept) 수행
```

#### 피디의 역할 상세

| 역할 | 세부 업무 | 빈도 |
|------|----------|------|
| **팀장** | 리뷰 작업 분배, 일정 관리 | 매일 |
| **품질 책임자** | 리뷰 기준 수립, 품질 메트릭스 관리 | 주간 |
| **기술 설계자** | React 아키텍처 설계, 컴포넌트 정의 | 수시 |
| **리포터** | 품질 리포트 PM 보고 | 주간 |

```
[작업 할당 Flow]

1. 사전 평가 (PM ↔ 시디/피디)
   ↓
2. 작업 분배 (시디)
   ↓

   ↓

   ↓

   ↓

   ↓

#### 상세 프로세스

**Step 1: 사전 평가 (작업 할당 전 30분 미팅)**

| 항목 | 내용 | 담당 | 산출물 |
|------|------|------|--------|
| 파일 분석 | 라인 수, 복잡도 확인 | 시디 | 사전 평가표 |
| 기술 검토 | 특이사항, 위험 요소 | 피디 | 리스크 체크리스트 |
| 일정 협의 | 예상 소요 시간 합의 | PM | 일정표 |

**Step 2: 작업 분배 (시디)**

```
[할당 기준]

- 시디: 복잡/초복잡 화면 (라인수 >20K 또는 WebSocket)
- 시니어: 복잡 화면 (라인수 15K~20K)
- 개발자: 일반 화면 (라인수 10K~15K)
- 주니어: 단순 화면 (라인수 <10K, 팝업)
```

**Step 3-5: 작업 수행**

| 활동 | 산출물 | 소요 시간 |
|------|--------|----------|
| 코드 분석 | 분석 노트 | 30% |
| 문서화 | 화면 분석서 | 40% |
| 자가 점검 | 체크리스트 완료 | 10% |
| 수정/보완 | 보완된 문서 | 20% |

**Step 6-8: 검토/완료**

| 단계 | 검토자 | 소요 시간 | 판정 |
|------|--------|----------|------|
| 시디 리뷰 | 시디 | 2시간 | Pass/수정 |
| 피어리뷰 | 피디/리뷰어 | 2시간 | Pass/수정/재검토 |
| 최종 완료 | PM | - | 완료 처리 |

```

   ↓

   ↓
