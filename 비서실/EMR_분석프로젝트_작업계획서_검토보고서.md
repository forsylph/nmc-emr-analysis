## 3. 팀 구조 검토

### 3.1 현황

**제안 구조**: 총 13명
- 개발팀: 10명 (시디1 + 시니어2 + 개발자5 + 주니어2)
- 피어리뷰팀: 3명 (피디1 + 시니어 리뷰어1 + 어시스턴스1)

### 3.2 검토 소견

| 평가 | 내용 |
|------|------|
| **구조적 장점** | • Phase별 점진적 투입 계획 (4명→13명)<br>• 역할 분담 세분화 (개발/리뷰 분리)<br>• 리더십 구조 명확 (시디/피디 팀장) |
| **현실성 문제** | • **13명 규모는 과도** - 분석 프로젝트 특성상 8~9명이 적정<br>• 개발자:리뷰어 비율 **10:3은 리뷰 리소스 과잉**<br>• 주니어 2명의 **산출물 품질 관리 부담** |

### 3.3 규모 적정성 분석

```
[인력 산정 검증]

총 화면 수: 20개
총 라인 수: 400,500라인
예상 소요: 42일

적정 인력 산정:
- 전체 작업량: 42일 × 8시간 = 336 시간
- 병렬 처리 시: 336시간 ÷ 42일 = 8명
- 리뷰 포함: 8명 + 리뷰 1명 = 9명

결론: 13명 → 9명으로 조정 가능
```

### 3.4 개선 권고사항

```diff
+ 권고 3-1: 팀 규모 축소 (13명 → 9명)
  
  개발팀: 7명 (시디1 + 시니어2 + 개발자3 + 주니어1)
  피어리뷰팀: 2명 (피디1 + 시니어 리뷰어1)
  
  근거:
  - 분석 프로젝트는 개발보다 문서 중심
  - 리뷰 리소스는 개발의 20-25%가 적정
  - Phase 1-2는 더 적은 인력으로 시작 가능

+ 권고 3-2: 역할 재조정
  - 시니어 개발자가 리뷰도 겸임 (2인 × 50%)
  - 피어리뷰팀 3명 중 1명은 외부 고문으로 전환
    → React 전문가 자문 (주 1회)

+ 권고 3-3: 주니어 개발자 운영 방식 변경
  - 단순 화면 분석만 담당 (2명 → 1명 축소)
  - 시니어와 페어로 작업 (멘토링 겸)
```

## 4. 일정 산정 검토

### 4.1 현황

**산정 공식**:
```
예상 소요시간 = 기본 시간 + (라인수 ÷ 5,000) × 복잡도 계수

- 기본 시간: 4시간
- 라인수 계수: 5,000라인당 2시간
- 복잡도 계수: 단순(0.8) / 일반(1.0) / 복잡(1.5) / 초복잡(2.0)
```

### 4.2 검토 소견

| 평가 | 내용 |
|------|------|
| **긍정적** | • 복잡도 기반 공식 **적용됨**<br>• 4단계 복잡도 분류가 현실적<br>• 산정 예시(4.2절)로 공식 검증 가능<br>• 10% 버퍼(5일) 포함 |
| **검증 필요** | • **공식의 검증 데이터 부재** (과거 프로젝트 기준?)<br>• 초복잡 화면(초록색)이 너무 많음 (9/20개)<br>• 리뷰 시간이 산정 공식에 미반영 |

### 4.3 산정 검증

| 화면ID | 라인수 | 복잡도 | 계산 | 계획 | 검토 의견 |
|--------|--------|--------|------|------|-----------|
| MMO00100 | 45,000 | 초복잡(2.0) | 22시간(2.8일) | 3일 | 적정 |
| SMPMI00300 | 30,000 | 초복잡(2.0) | 16시간(2.0일) | - | ⚠️ 2일은 과소 |
| SMPMO00300 | 28,000 | 초복잡(2.0) | 15.2시간(1.9일) | 2일 | ⚠️ 병원 핵심 업무, 3일 권고 |

### 4.4 개선 권고사항

```diff
+ 권고 4-1: 초복잡 화면 일정 재검토
  - 입원/처방 모듈(Phase 5): 15일 → 18일로 조정
  - 의료 안전 관련 화면은 추가 검토 시간 필요

+ 권고 4-2: 리뷰 시간을 산정 공식에 추가
  예상 소요시간 = 분석 시간 + (분석 시간 × 0.3)
  
  또는:
  - 단순: 리뷰 0.5일
  - 일반: 리뷰 1일
  - 복잡/초복잡: 리뷰 1.5일

+ 권고 4-3: 공식 검증 계획 수립
  - Phase 1(2일) 후 실제 소요 시간 측정
  - 공식 정확도 검증 및 보정
  - Phase 2 이후 공식 개선
```

### 5.2 검토 소견

| 평가 | 내용 |
|------|------|
| **우수** | • **6.1~6.3절**에 상세 체크리스트 존재<br>• 화면별 체크리스트(분석 시 필수 확인) 제공<br>• 위험도별 대안 기술 제시<br>• 5가지 주요 리스크(R1~R5) 식별 |
| **보완 필요** | • **밤샘 모드(5절)**가 리스크 관리의 주요 대응책으로 설정<br>• ⚠️ 밤샘은 근본 해결책이 아님<br>• React 구현 불가 시 **대안 기술 선정 기준** 미흡 |

### 5.3 밤샘 모드 우려사항

```diff
- 우려 1: 밤샘 모드가 지연 대응의 주요 수단으로 설정됨
  "대형 화면 지연 → 밤샘 모드 또는 인력 추가 투입"
  
- 우려 2: 건강/안전 원칙이 있으나 실행 가능성 의문
  - 3일 연속 밤샘 시 1일 휴식 규정
  - 실제 프로젝트에서는 지켜지기 어려움
  
- 우려 3: 품질 저하 가능성
  "피로 시 복잡한 작업 금지" - 실제로 통제 어려움
```

### 5.4 개선 권고사항

```diff
+ 권고 5-1: 밤샘 모드에서 "예방적 일정 관리"로 전환
  - 밤샘은 최후의 수단 (P0 이슈 시에만)
  - Phase별 20% 버퍼 시간 추가 (현재 10%)
  - 복잡 화면은 사전 POC(기술 검증) 수행

+ 권고 5-2: React 구현 불가 시 대안 기술 선정 기준
  | React 구현 어려움 | 대안 기술 | 선정 기준 |
  |------------------|-----------|-----------|
  | ActiveX | iframe 유지 | 기능 고유성 |
```
